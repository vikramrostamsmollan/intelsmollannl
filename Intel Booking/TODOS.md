## Pre-Production Gating Items (block rollout, not demo)

### TODO-1: Migrate Calendar refresh token to Google Secret Manager

**What:** Replace `firebase functions:config` storage of the Calendar service account refresh token with Google Secret Manager. Grant the Cloud Functions service account read access to the secret via IAM.

**Why:** Functions config is stored in plain text in project metadata, readable by anyone with `firebase functions:config:get` access to the project. Secret Manager provides access control, audit logging, and rotation support. Google refresh tokens for consumer accounts also expire after 6 months of inactivity with no warning — without rotation tracking, all booking confirmations silently fail after expiry.

**Pros:** Credential security, rotation support, audit trail.

**Cons:** Additional GCP service setup, ~1 hour to implement.

**Context:** For the prototype demo, Functions config is acceptable (small team, short window). Before production rollout, this is a hard requirement. The comment in `functions/index.js` will include a `// TODO: Secret Manager` note as a reminder.

**Depends on:** Milestone 1 complete (Firebase project + Calendar API configured).

## Phase 2 Features

### TODO-4: Rescheduling flow

**What:** Add a 'Wijzig boeking' (reschedule) option from the Dashboard. Flow: display available alternative slots → user selects new slot → Cloud Function deletes old Calendar event, creates new one, updates Firestore record, resets old slot.booked to false, sets new slot.booked to true.

**Why:** The primary pain point in the status quo (step 7) is that rescheduling restarts the entire coordination loop. Cancel + re-book is the current workaround. Full reschedule eliminates the last manual step.

**Pros:** Closes the biggest remaining workflow gap vs. the status quo. Turns the platform from "booking tool" to "full coordination replacement."

**Cons:** More complex transactional logic than cancel (two slots + one Calendar event update).

**Context:** Demo talking point: "Cancel and re-book for now — reschedule is the next feature on the roadmap." The cancellation flow (Milestone 4) is a direct prerequisite. The Firestore transaction pattern from /confirmBooking and /cancelBooking applies here too.

**Depends on:** Milestone 4 complete (cancellation flow working and tested).
